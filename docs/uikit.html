Shared poster cards used in templates/partials/library/video_list.html and assets/src/js/lib/discover/components/ItemGrid.jsx. The visual primitives are defined as CSS classes in assets/src/styles/style.css so both the Go-template and Preact implementations stay in sync.

Inception

Interstellar

2014
API surface:
  • .w-card-frame — card container. Auto-applies purple-themed border + hover. Has container-type: inline-size for container queries. Wrap inside .group parent (the link/clickable wrapper).
  • .w-card-frame.is-watched — modifier, sets opacity: 0.7 for watched items.
  • .w-card-title — card title. Auto-applies w-purpleL hover color when parent .group:hover.
  • .w-card-badge — overlay badge with text label. Add positioning (top-2 right-2) and color (text-green-400, text-yellow-400) inline.
  • .w-card-badge-label — wrap badge text in this class. Auto-hidden when card < 210px via container query to prevent badge overlap.
  • .w-card-badge-ghost — icon-only overlay, hover-revealed on desktop, opacity: 0.5 + enlarged to 2.5rem on touch devices. Add positioning + hover color (hover:text-green-400) inline.
  • .w-card-stars-compact — compact rating display (single star icon + number). Auto-hidden when card ≥ 210px.
  • .w-card-stars-full — full 5-star rating row. Auto-hidden when card < 210px.
Convention: watched = green (text-green-400) at top-right, rating = yellow (text-yellow-400) at top-left. Don't swap colors or sides — both Go template and Preact rely on this.
Container queries (210px threshold): .w-card-frame has container-type: inline-size. When the card is narrower than 210px, badge text labels (.w-card-badge-label) are hidden and .w-card-badge shrinks to icon-only padding. Stars switch from full (.w-card-stars-full) to compact (.w-card-stars-compact). This is card-width-based, not screen-width-based, so it adapts correctly regardless of the number of grid columns.

JavaScript API

import { initProgressLog } from "../../lib/progressLog";

// Initialize on a container element (must have .log-target inside)
const log = initProgressLog(document.querySelector('.my-alert'));

// Chain log entries
log
  .info('regular info message')               // gray dot, gray text
  .inProgress('task1', 'connecting...')      // cyan pulsing dot
  .updateStatus(42)                          // updates to "(42)" format
  .done('download complete')                 // cyan checkmark + summary row
  .warn('speed limited')                     // pink warning icon + summary row
  .error('connection failed')                // pink cross icon + summary row
  .close();                                  // reveals .alert-close-wrapper
Import from assets/src/js/lib/progressLog.js
Each method returns an Entry for chaining · inProgress(tag, message) creates new line · done/warn/error(message?) updates the tagged line and optionally adds summary row

14. Toast Notifications

Global toast notifications for user action feedback. Toasts appear in the bottom-right corner, auto-dismiss after 3 seconds, and stack when multiple are shown.

Types

Success toast
Info toast
Error toast

Usage (JavaScript)

Client-side triggers:
// Available globally via window.toast
toast.success('Copied!')
toast.error('Something went wrong')
toast.info('Download starting...')
Server-side triggers (automatic):
// In Go handlers, use web.RedirectWithSuccessAndMessage(c, "message")
// instead of c.Redirect(http.StatusFound, ...)
// This appends ?status=success&message=... to the redirect URL.
// The async fetch wrapper detects these params and shows a toast.
web.RedirectWithSuccessAndMessage(c, "Domain added")
web.RedirectWithSuccessAndMessage(c, "Settings saved")

CSS Classes

.toast-alert — base class
.toast-alert-success / .toast-alert-info — cyan variant (matches btn-soft-cyan)
.toast-alert-error — pink variant (matches btn-soft)
Rules:
• Success/Info toasts use cyan colors — consistent with progress-alert done states
• Error toasts use pink colors — consistent with progress-alert error states
• Toast container is #toast-container with DaisyUI classes toast toast-end toast-bottom z-tooltip
• Use toast.success() for completed actions (copy, save, add, delete)
• Use toast.info() for started actions (download starting)
• Use toast.error() for client-side failures

15. Mobile Patterns

Rules and patterns for mobile-optimized layouts. These ensure pages stay compact on small screens (320px–480px) while preserving full content on desktop.

Compact Page Headers

On mobile, decorative elements waste valuable vertical space. Hide non-functional elements to get content above the fold faster.

  • Decorative badges (e.g. "LIBRARY", "DISCOVER"): hidden sm:inline-flex
  • Subtitle / description text: hidden sm:block
  • Top padding: pt-24 sm:pt-[120px] (96px on mobile vs 120px on desktop — still clears the 72px navbar)
  • Heading margin: mb-2 sm:mb-4

Sticky Tab Bar

Tab navigation sticks below the navbar on scroll so users can switch sections without scrolling up.

Pattern:
sticky top-[72px] z-10 -mx-3 sm:-mx-6 px-3 sm:px-6 py-3 bg-w-bg/90 backdrop-blur-lg border-b border-w-line/30

  • Negative horizontal margin (-mx-3) stretches the bar edge-to-edge, then px-3 restores inner padding
  • Use flex-col sm:flex-row inside the bar: tabs on first row, filters/sort on second row on mobile; all in one row on desktop
  • Filters/sort get sm:ml-auto to push right only on desktop

Navbar at Narrow Widths (320px)

At 320px the navbar is very tight with logo + 5 buttons. Use tighter padding and gaps at small widths.

  • Navbar padding: px-2 min-[400px]:px-3 sm:px-6
  • Navbar-end gap: gap-0.5 min-[400px]:gap-1 sm:gap-2
  • Don't hide nav buttons — all are important. Shrink gaps instead.

Touch Targets

Interactive elements on touch devices need larger hit areas.

  • .w-card-badge-ghost enlarges from 2rem to 2.5rem via @media (hover: none)
  • Minimum recommended touch target: 40px (2.5rem). Apple guidelines: 44pt.

Container Queries for Cards

Card content adapts based on the card's own width (not the viewport), via container-type: inline-size on .w-card-frame. This handles all grid configurations automatically.

Threshold: 210px

Below 210px (compact mode):
  • .w-card-badge-label — hidden (badge becomes icon-only)
  • .w-card-badge — padding shrinks to 0.375rem (square pill)
  • .w-card-stars-full — hidden
  • .w-card-stars-compact — visible (single star icon + number, e.g. "★ 7.6")
At or above 210px (full mode):
  • .w-card-badge-label — visible (badge shows text)
  • .w-card-stars-full — visible (5-star row)
  • .w-card-stars-compact — hidden
Why container queries, not media queries? The same card can appear in a 2-col grid (wide cards) on mobile and a 4-col grid (narrow cards) on desktop. Media queries would show full stars on desktop even when cards are narrow. Container queries adapt to actual card width.

Torrent List Names

  • Use line-clamp-2 instead of truncate for torrent file names — shows more of the name on narrow screens with minimal height increase

17. Discover Watchlist

The Discover page exposes a per-user "save for later" surface. Two visual elements drive it: a card-level heart toggle and a page-level mode switcher (Catalog | Watchlist). Catalog uses the discover cyan accent; Watchlist uses a pink (w-pink / w-pinkL) reserved exclusively for this family. Brand pink stays CTA-only.

Heart Badge (card corner)

Lives in the bottom-right of every IMDB card across catalog, search, AI recommendations and the stream modal poster. Top corners are taken by the green WatchedBadge (top-right) and yellow RatingBadge (top-left); pink heart at the bottom keeps the four-quadrant layout legible. Filled heart = saved, outlined heart with hover-to-red = not saved.

Mode Switcher (sticky bar, right side)

Two-button join group anchored to the right of the type tabs. Catalog uses the cyan chipClass family (matches type/search tabs and the rest of discover). Watchlist uses the pink Watchlist family. Each button always hover-lights into its own colour so the user reads "I will activate this side".

Status Pill (modal join group)

Inside the stream modal the Watchlist toggle joins a three-button group with Watched (green) and Rated (yellow). Each follows the same shape: ghost with line border + coloured hover when off, ghost with thin coloured border + hover-fill when on. Sibling colours stay distinct (green / yellow / pink) so the same join group can hold all three states without ambiguity.

Off (any): btn btn-ghost btn-xs join-item border border-w-line text-w-sub hover:border-{color}/40 hover:text-{color}
On (any): btn btn-ghost btn-xs join-item border border-{color}/20-30 text-{color} hover:bg-{color}/10 hover:border-{color}/40
Colours: green-500/green-400 (Watched), yellow-500/yellow-400 (Rate), w-pink/w-pinkL (Watchlist).
Rules:
• Pink (w-pink/w-pinkL) is shared between brand CTAs (btn-pink) and the Watchlist family. The two contexts don't visually collide because Watchlist always sits inside a chip / heart / join group with its own border-and-fill pattern, never a solid CTA.
• Mode switcher: Catalog hover-lights cyan, Watchlist hover-lights pink. Each side stays in its own colour family even when inactive so the left/right pair always reads correctly.
• Badge corner mapping is fixed: green top-right (Watched), yellow top-left (Rated), pink bottom-right (Watchlist). Changing positions breaks card scan order.
• Active modal pills: btn-ghost border border-{color}/20-30 text-{color} hover:bg-{color}/10. Inactive: btn-ghost border border-w-line text-w-sub hover:border-{color}/40 hover:text-{color}. Adding a new state colour means picking a hue distinct from green / yellow / pink.
• Helpers: chipClass() (cyan, type/search tabs), catalogChipClass() (cyan, mode switcher Catalog), watchlistChipClass() (pink, mode switcher Watchlist). All live in assets/src/js/lib/discover/components/discoverUtils.js — don't open-code the chip classes.
