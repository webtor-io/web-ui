# Internationalization (i18n) — Technical Guide

## Overview

web-ui supports 7 languages: English (default), Russian, Spanish, German,
French, Portuguese (PT-BR), Italian. English has no URL prefix (`/`), others
use path prefix (`/ru/`, `/es/`, `/de/`, `/fr/`, `/pt/`, `/it/`).

**Note on Portuguese**: the `pt` bundle carries Brazilian Portuguese (PT-BR)
content. We use the bare two-letter code for URL/middleware simplicity — the
HTTP middleware assumes 2-char prefixes (see `services/i18n/middleware.go`).
Accept-Language `pt-BR` / `pt-PT` both normalise to `pt` via `language.Base()`
in `detectBrowserLang`, so Brazilian browsers auto-route here. If/when we add
European Portuguese, split this into `pt-br` + `pt-pt` and teach the middleware
about variable-length prefixes.

## Architecture

```
HTTP Request
  → i18n HTTP Middleware (strips /ru/ prefix, sets X-Lang header, manages lang cookie)
  → Gin Router (routes unchanged, no per-language registration)
  → i18n Gin Middleware (reads X-Lang, stores lang + Localizer in gin.Context)
  → Handler (creates web.Context with Lang field)
  → Template (uses helper functions for translation)
```

### Key Files

| File | Purpose |
|------|---------|
| `locales/{en,ru,es,de,fr,pt,it}.json` | Translation files (flat key-value JSON) |
| `locales.go` | Embeds locale files via `//go:embed` |
| `services/i18n/i18n.go` | Service: bundle, localizer factory |
| `services/i18n/middleware.go` | HTTP middleware: prefix stripping, redirect, cookie |
| `services/i18n/gin_middleware.go` | Gin middleware: lang context resolution |
| `services/i18n/helper.go` | Template helpers: `t()`, `tp()`, `TranslateWithLocalizer()` |
| `handlers/i18n/handler.go` | RegisterHandler: wires HTTP + Gin middleware |
| `services/web/context.go` | `web.Context.Lang` field, `Path()` method |
| `services/web/helper.go` | URL helpers: `langPath`, `langURL`, `hrefLangs`, `withContext`, `blogLang` |
| `services/template/template.go` | FuncMap applied before LayoutBody parse |

## Template Patterns

### 1. Simple translation

```html
{{ t $.Lang "nav.discover" }}
```

### 2. Translation with parameters

```html
{{ tp $.Lang "footer.copyright" "Year" now.UTC.Year }}
```

Locale file: `"footer.copyright": "Copyright © {{.Year}} Webtor.io"`

### 3. Internal links — always use `langPath`

```html
<a href="{{ langPath $.Lang "/discover" }}">...</a>
```

On `/ru/` renders as `/ru/discover`. On `/` renders as `/discover`.

### 4. Sub-templates receiving `*web.Context`

When the sub-template is called with `$` or `.` where data is `*web.Context`:

```html
{{ template "nav" $ }}
{{ template "footer" $ }}
```

Inside the sub-template: `{{ $.Lang }}`, `{{ t $.Lang "key" }}` work directly.

### 5. Sub-templates inside `{{ with .Data }}`

Inside `{{ with .Data }}`, `.` becomes the Data struct, but `$` still refers to root `*web.Context`.

**Rule:** When calling sub-templates from `{{ with .Data }}` block, pass `$` not `.`:

```html
{{ with .Data }}
    {{ template "about" $ }}          ✅ passes *web.Context
    {{ template "about" . }}          ❌ passes *index.Data — $.Lang won't work
{{ end }}
```

Inside the sub-template, access data fields via `$.Data.FieldName` instead of `.FieldName`.

### 6. Sub-templates receiving non-Context data — `withContext`

Partials should receive only the data they need, not the full `*web.Context`.
Use `withContext` to wrap specific data with context:

```html
{{ template "button" withContext $ (makeFileDownload $ .) }}
{{ template "vault/button" (withContext $ .VaultButton) }}
{{ template "library/button" (withContext $ .Resource) }}
{{ template "library/sort" (withContext $ .Data.Args) }}
{{ template "profile/vault" (withContext $ .Data.VaultStats) }}
```

Inside the sub-template:
- `{{ .Ctx.Lang }}` — language code (outside `range`/`with` blocks)
- `{{ $.Ctx.Lang }}` — language code (inside `range`/`with` blocks where `.` changes)
- `{{ .Ctx.User | hasAuth }}` — full context access
- `{{ .Data.Name }}` — domain data
- `{{ t .Ctx.Lang .Data.Name }}` — translate domain data field

**⚠ Inside `{{ with .Data }}` or `{{ range .Data }}`, `.` changes!**
Use `$.Ctx.Lang` (with `$` prefix) to access context:

```html
{{ define "library/torrent_list" }}
<div>
    {{ range $.Data }}
        <a href="{{ langPath $.Ctx.Lang ... }}">  ← $ required inside range
            {{ t $.Ctx.Lang "library.files" }}
        </a>
    {{ end }}
</div>
{{ end }}
```

**When to use `withContext`:**
- Partial needs only 1-2 fields from Data + Lang → pass specific field
- Partial is a focused component (button, badge, modal, list) → narrow scope
- Partial sits next to other `withContext` components → consistency

**When full context is OK:**
- Partial needs User/Claims for auth checks + multiple Data fields
- Partial is a large container (file browser, nav, footer)

**Never add Lang to domain data structs.** Use `withContext` instead.

The `CtxData` struct is defined in `services/web/helper.go`:
```go
type CtxData struct {
    Ctx  *Context
    Data any
}
```

### 7. HTML in translations — `tHTML` / `tpHTML`

Some translations contain trusted HTML (links, `<strong>`, etc.). Since `t`/`tp`
return `string` and Go's `html/template` auto-escapes strings, use `tHTML`/`tpHTML`
which return `template.HTML`:

```html
{{/* Simple — no parameters */}}
{{ tHTML $.Lang "instructions.stremio.step3" }}

{{/* With parameters — e.g. dynamic URL */}}
{{ tpHTML $.Lang "instructions.stremio.step1" "ProfileURL" (langPath $.Lang "/profile") }}
```

Locale file example:
```json
"instructions.stremio.step1": "Copy your addon link from your <a href='{{.ProfileURL}}' class='text-w-pinkL hover:underline'>Webtor profile</a>."
```

**When to use:** Only for content-heavy pages (instructions, about) where inline
HTML is unavoidable. For regular UI strings, prefer `t`/`tp` with HTML in the template.

**Security:** Translation files are committed to git and controlled by the dev team.
Never use `tHTML`/`tpHTML` with user-supplied content.

### 8. Translation keys in Go code

When Go code provides display strings (tool titles, sort labels, button names),
use i18n message keys instead of English strings:

```go
// handlers/common/tools.go
{Url: "torrent-to-ddl", Title: "tool.torrentToDdl.title", ...}

// models/library.go — sort options
case SortTypeRecentlyAdded:
    return "library.sort.recentlyAdded"

// handlers/resource/helper.go — button names
s.MakeButton(ctx, gd, "resource.stream", "stream", "/stream-video")
```

Template resolves: `{{ t $.Lang .Title }}`, `{{ t .Ctx.Lang .Data.Name }}`

### 8. Blog links

Blog supports only EN and RU. Use `blogLang` helper:

```html
<a href="https://blog.webtor.io/{{ blogLang $.Lang }}/post/">Blog</a>
```

### 9. Form actions

POST form actions (`/lib/add`, `/vault/pledge/add`, etc.) do NOT need `langPath`.
They are server-side endpoints; the redirect after POST uses `X-Return-Url`
which the async JS sets to the current page URL (already with lang prefix).

### 10. `data-async-layout` strings

These inline template strings are parsed at runtime. They have access to all
registered FuncMap functions (including `withContext`, `t`, `langPath`).
This works because FuncMap is applied before LayoutBody parsing in `template.go`.

```html
data-async-layout="{{`{{ template "vault/button" (withContext $ .Data.VaultButton) }}`}}"
```

### 11. Job messages (Go backend)

Job scripts generate progress messages shown to users. These are translated
using `i18n.Service` passed through the `Jobs` struct.

**Pattern — `Jobs.T()` for inline scripts:**
```go
func (s *Jobs) T(c *web.Context, key string) string {
    return i18n.TranslateWithLocalizer(s.i18n.Localizer(c.Lang), key)
}

// Usage:
j.InProgress(s.T(c, "job.retrievingData"))
j.Redirect("/"+rID, s.T(c, "job.redirecting"))
```

**Pattern — `script.t()` for dedicated script structs:**
```go
// ActionScript, LoadScript, EmbedScript each have:
func (s *ActionScript) t(key string) string {
    return i18n.TranslateWithLocalizer(s.i18n.Localizer(s.c.Lang), key)
}

// Usage:
j.InProgress(s.t("job.retrievingStreamUrl"))
j.DoneWithMessage(s.t("job.downloadReady"))
```

`i18n.Service` is passed: `Jobs` → `scripts.Action()` / `scripts.Load()` / `scripts.Embed()` → script structs.

## URL Strategy

| URL | Language | Notes |
|-----|----------|-------|
| `/` | English | Default, no prefix, backward compatible |
| `/ru/` | Russian | |
| `/es/` | Spanish | |
| `/de/` | German | |
| `/fr/` | French | |
| `/pt/` | Portuguese (PT-BR) | Brazilian Portuguese content under the bare `pt` code |
| `/it/` | Italian | |
| `/en/*` | → 301 to `/*` | Canonical English has no prefix |

### Auto-redirect (first visit)

When no `lang` cookie exists, the HTTP middleware checks `Accept-Language`:
- If preferred language is supported and non-English → 302 redirect + set cookie
- If English or unknown → set cookie `lang=en`, serve as English

Cookie `lang` is also set when visiting any prefixed URL (`/ru/...`).

### Bare-path honours cookie preference

When a logged-in user hits a bare (non-prefixed) path like `/auth/callback`
(external OAuth redirect) or a bookmarked `/discover`, and the `lang` cookie
holds a non-default value, the middleware 302-redirects to `/{cookie}/path`
so the user's stored preference survives external entry points.

Only safe methods (`GET`/`HEAD`) are redirected — `POST`/`PUT`/`DELETE`/`PATCH`
form actions are passed through untouched so their bodies are not dropped.

### Explicit English switch — `?lang=en`

The language switcher's English entry is rendered as `/foo?lang=en` (not
bare `/foo`). The `?lang=X` query is an explicit user choice: the middleware
sets `cookie=X`, strips the query and 302-redirects to the canonical URL for
that language. Without this signal the bare-path rule above would bounce a
user with `cookie=ru` back to `/ru/foo`.

### SEO

- `<html lang="{{ $.Lang }}">` — dynamic
- `<link rel="alternate" hreflang="...">` — via `{{ hrefLangs $.Path }}`
- `<link rel="canonical">` — via `{{ canonicalURL $.Lang $.Path }}`
- `sitemap.xml` includes `<xhtml:link>` alternates for all languages

## Translation Guidelines

### General Rules

1. **Don't translate brand names**: Vault, Discover, Stremio, Webtor, WebDAV
2. **Don't translate technical terms**: DDL, ZIP, HLS, MKV, VPN, HTTPS, API
3. **Legal pages stay in English** — TOS, Privacy Policy, DMCA are not translated
4. **Keep translations natural** — don't transliterate English words unnecessarily

### Russian-Specific

| Avoid | Use instead | Reason |
|-------|-------------|--------|
| Стримить/Стримьте | Смотрите | "Стримить" sounds unnatural in Russian |
| Залог (vault pledge) | Вклад | "Залог" has negative connotation (collateral/bail) |
| Стриминг | Просмотр / Воспроизведение | Where context allows, keep in technical contexts |
| Транслитерация | Русский эквивалент | Unless no Russian word exists |

### Portuguese (PT-BR) specific

- **Target audience is Brazilian**. Use BR spelling and idioms ("assistir", not "ver"; "você", not "tu"; "arquivo", not "ficheiro"; "baixar", not "descarregar"; "celular", not "telemóvel").
- **Anglicisms are fine** for tech terms that everyone uses in EN in Brazil ("stream", "streaming", "seeder", "cache", "buffer", "addon", "player"). Don't force literal translations.
- **Informal tone**: Webtor's voice is friendly; use "você", contractions like "pra" in casual CTA strings ("Bora testar?"). Keep formal register for legal/privacy content.

### French-Specific

- Use **"vous"** (formal address) throughout — French web UX convention, feels natural to the full audience (FR, BE, CH, QC). Informal "tu" would feel jarring in a product interface.
- Non-breaking spaces before `:`, `!`, `?`, `;` are typographically correct in French but omitted here to keep JSON values simple and avoid breaking copy/paste tooling.
- **Don't translate** "streaming" (widely used in French tech speak). "Diffuser" is the verb form, "streaming" stays as the noun.

### Italian-Specific

- Use **"tu"** form (Italian web UX convention for consumer products).
- **Don't translate** "streaming", "player", "seeder", "peer", "cloud", "addon" — all are standard in Italian tech context.
- "Bandwidth" → "banda" in casual contexts, "larghezza di banda" when formality helps.

```
section.subsection.element

nav.discover              — navigation items
footer.tools              — footer sections

```

## Adding a New Language

1. Create `locales/{code}.json` — copy from `en.json`, translate values
2. Add code to `services/i18n/i18n.go`: `SupportedLangs` and `LangNames`
3. Done — middleware, routing, switcher pick it up automatically

## Locale File Format

Flat JSON, key → string. For parameterized strings use Go template syntax:

```json
{
    "footer.copyright": "Copyright © {{.Year}} Webtor.io",
    "about.faq.aiQuota.a": "Every account gets {{.FreeQuota}} free AI requests per day."
}
```

Resolved in templates via `{{ tp $.Lang "key" "Param" value }}`, and in Go via
`TranslateWithLocalizerData(loc, key, map[string]any{"Param": value})` (see
`ActionScript.tp` for a script-scoped wrapper).

### Client-side placeholders (`%v`)

The client-side `tf()` helper in `assets/src/js/lib/i18n.js` is a thin
`String.replace('%v', arg)` loop — it does NOT understand Go template syntax.
For JS-only keys (e.g. `discover.addonStreams`) keep `%v` placeholders in the
locale files and call `tf('key', arg1, arg2)`. Keys consumed by both Go and JS
should use Go template syntax and only be called from Go via `tp` / `Tp`.

## Async Page Updates

Language switching uses `data-async-target="main"`. The layout updates:
- `<main>` content — via standard async mechanism
- `<nav>` — via `data-async-update-nav`
- `<footer>` — via `data-async-update-footer`
- `<title>` and `<meta description>` — via `data-async-update-title` / `data-async-update-description`

Handler: `assets/src/js/app/layout.js` — `update()` function processes all `data-async-update-*` attributes.
